# Vanish Tac Toe

3つまでしか置けない新感覚マルバツゲーム。4つ目を置くと最も古い駒が消えるという消滅ルールの下で、**到達可能な 128,170 状態すべてを事前に完全解析**し、全局面の最善手と勝敗予測をクライアント側で即座に参照できるようにしたゲーム兼ゲーム理論可視化ツール。

**Live Demo**: https://9-vanishtactoe.pages.dev/

---

## プロジェクトの特徴

- **完全解析済みゲーム** — レトログレード解析（BFS + 逆方向評価伝播）で到達可能な全 128,170 状態を事前計算し、ソルバをクライアントに埋め込み
- **完全情報の可視化** — 通常のゲームAIは「探索して最善手を出す」だけだが、このアプリは全マスに勝敗（勝ち/引き/負け）と残り決着手数をオーバーレイ表示し、ゲーム木の断面をUIとして提示
- **3段階の難易度AI** — 完全解析データを基盤にした確率的ランダム化で、直感的に強さを制御できる AI を実装
- **ゼロランタイムソルバ** — ランタイムのソルバ計算は `Map<stateKey, evaluation>` の O(1) ルックアップのみ、サーバー不要
- **ランニングコスト 0 円** — Cloudflare Pages の無料枠で運用

## コアアルゴリズム: レトログレード解析

通常の Minimax は再帰探索を前提としますが、このゲームは**消滅ルールにより同じ局面が循環的に再出現する**ため、単純な再帰探索は停止しません。代わりに以下の 2 段階で解析しています（チェスのエンドゲームテーブルベースと同じ手法）。

### Step 1: 状態空間の BFS 列挙

空の盤面からすべての合法手を辿り、到達可能な状態を幅優先で列挙します。状態は `(board, q0, q1, turn)` のタプルでハッシュ化してユニーク判定。

- `board`: 9 マス × `{空, P1, P2}` の 3 状態
- `q0` / `q1`: プレイヤーごとの配置順序キュー（FIFO）
- `turn`: 手番

**結果: 128,170 ユニーク状態**

### Step 2: 逆向き評価伝播

ゲーム終了局面（3つ揃い／手詰まり）から逆方向に親状態を辿り、次の規則で評価を確定させます：

- 自分の手番で**勝ち子局面が1つでも存在** → その状態は **勝ち**
- 自分の手番で**全子局面が負け** → その状態は **負け**
- 上記どちらでもない（循環含む） → **引き分け**

副産物として各局面の**最短決着手数（depth）**も同時に記録し、AI が「負けるにしても粘る」「勝つなら最短で」といった挙動を実現しています。

### 解析結果

| 区分 | 状態数 |
|------|-------:|
| 先手必勝 | 54,853 |
| 後手必勝 | 60,124 |
| 引き分け | 13,193 |
| **合計** | **128,170** |

数学的には先手必勝ですが、**3手目以降は毎回勝ち手がちょうど1つしか存在しない**ため、人間が実際に完璧な手順をたどって勝つのは約 0.14%（(1/3)^6）という発見も得られています。

## 実装のハイライト

### 状態遷移を単一の純関数に閉じ込める

`applyMoveToState(state, cell)` が FIFO による駒消滅・新駒配置・勝敗判定・手番交代を一つの純関数で処理。React の `setGame(prev => ...)` と組み合わせることで、StrictMode の二重実行下でも破綻しません。

```ts
function applyMoveToState(state: GameState, cell: number): GameState {
  // プレイヤーの駒が MAX_PIECES 以上なら FIFO で最古駒を除去
  // 新駒配置 → 勝敗判定 → 次の手番を返す
  ...
}
```

### 完全解析データを基盤にした確率的難易度AI

評価関数を手でチューニングする代わりに、**完全解析の結果を基盤にして確率的ランダム化を重ねる**ことで、難易度の「強さ感」を直感的に制御しています。

```ts
function pickAiMove(evaluation, turn, difficulty): number {
  if (difficulty === 'hard') return pickBestCell(evaluation, turn)
  const p = difficulty === 'easy' ? 0.25 : 0.65
  return Math.random() < p ? pickBestCell(evaluation, turn) : randomPick()
}
```

- **上級**: 100% 最適手 — チキンゲーム的な膠着に持ち込まれ、ほぼ勝てない
- **中級**: 65% 最適 + 35% ランダム — 時々ミスを拾える
- **初級**: 25% 最適 + 75% ランダム — 自分のリーチすら見逃すレベル

`pickBestCell` は `(自分の勝ち) > (引き分け) > (自分の負けだが粘る)` の順でスコアリングし、勝ち/負けの場合は `depth` でタイブレーク。

### 「次に消える駒」の視覚化

FIFO の最古駒を取得し、自分の手番の最古駒には**赤い脈動バッジ + 駒自体の点滅**、相手の最古駒には**地味なグレーバッジ**を付与。消滅ルールの「次に何が起きるか」が一目でわかります。

### アクセシビリティ

- `role="grid"` + セル単位の詳細な `aria-label`（例: 「中央 ほのお 次に消える駒」）
- `aria-live="polite"` で手番/勝敗の状態変化を読み上げ
- `prefers-reduced-motion: reduce` で全アニメーション停止

### パフォーマンス最適化

- `useMemo` で評価データを手番・盤面変更時のみ再計算
- 事前計算データのルックアップは `Map` で O(1)
- フォント `preconnect` でサードパーティロード時間を短縮

## ゲーム理論的な発見

完全解析の過程で、このゲーム特有のいくつかの現象が観察できました（アプリ内「学ぶ」ページで詳述）：

- **理論と実践の乖離**: 数学的には先手必勝だが、勝ち筋が 3 手目以降毎回 1 つしかないため、実質的には実行不可能
- **チキンゲーム構造**: 両者がリーチ防御 + 中央・角優先で打つと **100% 千日手**。膠着を崩した側が約 80% 負ける
- **ヒューリスティックの失敗**: 「中央は強い」「角を取れ」という通常の三目並べの経験則が、消滅ルール下では完全に裏切られる

小さな 3×3 の盤面から、ゲーム理論の教科書に載るような構造がいくつも観察できる——というのがこのプロジェクトの副産物です。
